Ramboll Developer Hub Concept Document
1 Product Vision
A Ramboll Developer Hub: one internal place where developers, data engineers, platform teams, architects,
DevOps, and product teams can discover systems, understand ownership, follow approved engineering
patterns, launch new projects, access documentation, and automate routine work.
The hub should not be like “another documentation site.” It should be a developer operating system
for Ramboll.
• Find what exists.
• Understand how it works.
• Start new work correctly.
• Connect delivery, code, knowledge, and governance in one place.
2 Structure — Home
A clean sidebar organized around developer intent, not internal team structure. The home dashboard is the
entry point and must give users a reason to return daily.
Area Content
My work Assigned issues, pull requests, reviews, incidents, deployments
My services Repositories, APIs, components, data products, cloud resources
Recommended actions Outdated docs, failing checks, missing ownership, security alerts
Recently viewed Repos, services, docs, project pages
Announcements Platform changes, upcoming migrations, new standards
Quick actions Create service, request access, open support ticket, ask platform
agent
Why this matters
A core pain point for a DPE Engineer is working across multiple projects simultaneously while
maintaining many older projects. The context-switching overhead here is significant and represents a
clear opportunity to streamline day-to-day work through a unified, personalized home view.

Score category Measures
Documentation README, ownership, runbook, API docs

Reality check
Projects today are scattered across GitLab, GitHub, Azure DevOps, and Bitbucket — fragmented
and low on transparency. The hub should unify visibility without removing the ability to keep invite-only
repositories where a security clearance is genuinely required (critical infrastructure, military, etc.).

8 Ramboll Design Language Direction
The interface should feel professional, calm, engineered, and trustworthy — leaning into clarity, sustain-
ability, engineering confidence, and human-augmented workflows.
8.1 Design principles
Principle UI expression
Clarity first Clean layouts, strong hierarchy, minimal clutter
Engineering precision Structured cards, status labels, metadata tables
Human-centered Helpful empty states, guided flows, plain language
Sustainable digital Avoid dashboard overload, prioritize useful signals

workflows to reduce friction across the software lifecycle.
